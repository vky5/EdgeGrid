# Token & secret inventory

Every credential in the system, where it lives, how it's generated, and what
happens if it leaks. Cross-reference with `auth-architecture.md` for the
flows these fit into.

| Secret | Where stored | Generated by | Read by | Blast radius if leaked |
|---|---|---|---|---|
| `NEXTAUTH_SECRET` | `web/.env.local` (server env) | You, via `openssl rand -base64 32` | NextAuth internals only | Forge any user's session JWT — full impersonation of any GitHub-linked user, including admin. Highest-value secret in the whole system. |
| `GITHUB_CLIENT_ID` / `GITHUB_CLIENT_SECRET` | `web/.env.local` | GitHub OAuth App settings | NextAuth GitHub provider | Attacker could stand up a phishing OAuth flow under your app's identity, or (with secret) exchange codes for tokens. Doesn't by itself forge sessions — still needs `NEXTAUTH_SECRET`. |
| `COORDINATOR_ADMIN_TOKEN` (gateway token) | `web/.env.local` (Next.js side); `data/admin.token` (coordinator side) | `nodeident.RandomToken(32)` in `internal/agent/agent.go:157`, printed once to stdout on first boot | `coordFetch()` (`web/lib/coordinator.ts:15`) attaches it; `requireGateway()` (`internal/coordinator/api.go:76`) checks it | **Full coordinator API access** — submit/cancel any job, approve/reject any node join, read/download any job's data and checkpoints, list all workers. Bypasses all per-user ownership checks, since those only exist in Next.js. This is the single point where "backend trust" lives. |
| `node.token` (per-node NATS password) | `node_auth` KV bucket (coordinator); `data/node.token` (on the node itself) | `nodeident.RandomToken(32)` in `handleJoinApprove` (`api.go:671`) | Coordinator (to add the NATS user); the node itself | Impersonate that one worker/server on NATS — publish fake heartbeats/results, receive training jobs meant for it, read/write whatever subjects its role permits. Scoped to one node, not the whole system. |
| `cluster.secret` | `data/cluster.secret` on each server-role node | `nodeident.RandomToken(32)` in `internal/agent/agent.go:75`, generated by whichever server node boots first / needs one | Any node joining as a `server` (cluster peer); NATS `Cluster.Password` | **Join the NATS cluster as a full route peer**, not just a client — this is a much bigger grant than a worker token. A leaked cluster secret lets an attacker's NATS node peer into your Raft cluster, see all replicated JetStream data, and potentially participate in consensus. Single shared secret for *all* peers (not per-node), so it can't be revoked for just one bad actor without rotating it everywhere. |
| `coord.secret` | `data/coord.secret` | `nodeident.RandomToken(32)` in `agent.go:65-69` | Loaded by `coordinator.go:115` as the coordinator's own privileged NATS client credential | Impersonate the coordinator's own NATS client connection — read/write anything the coordinator process itself can. |
| `node.id` | `data/node.id` | `crypto/rand`, 16 bytes hex, in `nodeident.LoadOrCreate` (`ident.go:22`) | Not a secret by design — used as the NATS *username* and the join-request KV key | Not sensitive on its own (128 bits, unguessable), but it functions as a de facto bearer key for `GET /join/{nodeID}` post-approval — see `known-gaps.md` #2. Treat it as semi-sensitive once a node is approved. |
| `ADMIN_GITHUB_USERNAME` / `NEXT_PUBLIC_ADMIN_GITHUB_USERNAME` | `web/.env.local` | You set it manually | `isAdmin()` (`web/lib/auth.ts:35`) | Not a secret (it's a username, and the `NEXT_PUBLIC_` copy is shipped to the browser on purpose for UI toggling). Authorization decisions use the server-only copy, so this can't be spoofed by editing client-visible env — only by controlling the server's actual environment. |

## Where secrets live on disk

All of the coordinator/node-side secrets above land in the process's
`--data-dir` (default `data/`) as flat files, written 0600 via
`nodeident.SaveToken` (`internal/nodeident/ident.go:68`):

```
data/
  admin.token      # gateway token (coordinator)
  coord.secret     # coordinator's own NATS client credential
  cluster.secret   # NATS cluster-route shared password
  node.token       # this node's NATS client credential
  node.id          # this node's identifier (not secret, but semi-sensitive)
  nats/            # JetStream storage — gitignored today
```

**Only `data/nats/` is gitignored right now.** The five files above are not
— see `known-gaps.md` #1, this is a live gap, not a hypothetical.

## Rotation story

There isn't one, currently. Every secret above is generate-once,
persist-to-disk, reuse-forever. No expiry, no rotation tooling, no way to
revoke a single compromised node's `node.token` without manually editing the
`node_auth` KV and hot-reloading NATS users. Worth keeping in mind if a
laptop or disk ever gets shared/reimaged without wiping `data/` first.
